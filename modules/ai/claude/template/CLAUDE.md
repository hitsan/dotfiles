# Claude Instructions

## Language
- 常に日本語で回答する。コード・識別子・技術用語はそのまま英語で書く。

## Response style
- 回答は短く構造化する（見出し・箇条書き）。
- 末尾の要約は不要。diff を見れば分かる。
- 探索的な質問（「どうすべきか」「何が考えられるか」）には 2〜3 文で答え、実装はユーザーの承認後に行う。

## Behavior
- 前提が欠けているときだけ質問する。それ以外は合理的に仮定し、仮定を一言明示する。
- 指示されたことだけ実装する。関係ない部分は触らない。
- リファクタや抽象化はタスクが要求する範囲を超えない。

## Code
- コメントはデフォルト不要。WHY が自明でない場合のみ 1 行書く。
- エラーハンドリングは実際に起こりうる境界（ユーザー入力・外部 API）のみ。
- セキュリティ脆弱性（インジェクション・XSS など）を書いたら即座に直す。

## TDD（t_wada スタイル）

実装タスクは常に TDD サイクルで進める。

### 進め方
1. **テストリストを作る** — 実装前に「何をテストすべきか」を箇条書きで列挙する。これがロードマップになる。
2. **Red** — テストリストから 1 つ選び、失敗するテストを書く。コンパイルエラーも Red。
3. **Green** — テストを通す最小限のコードを書く。きれいさは不要。
   - *仮実装*: まず定数を返すだけでよい。
   - *三角測量*: 例を増やして汎化を強制する。
   - *明白な実装*: 答えが明白なら直接書く。
4. **Refactor** — Green を保ったままコードを整理する。テストコードも対象。
5. **繰り返す** — テストリストの次の項目へ。

### ルール
- 1 サイクルのステップは小さく保つ。Red → Green → Refactor の順を崩さない。
- Refactor は必ず Green の状態で行う。Red のまま整理しない。
- テストが通ったらコミットのタイミング。
- テスト中に新たな懸念が生まれたらテストリストに追記し、今のサイクルは完了させる。

## Safety
- 破壊的・不可逆な操作（force push, rm -rf, DB drop など）は実行前に確認する。
- シークレット・機密情報をコードやログに含めない。
