# HTML Report

自由形式の内容を、単一の自己完結HTMLファイルに整形してローカルに保存する。
固定のテンプレートは持たない。内容ごとに構成もコンポーネントも都度選ぶ。

## ワークフロー

### Step 1: 内容を理解し構成を決める

渡された内容（調査結果・レビュー結果・作業ログ・分析結果など）から以下を
組み立てる。全部揃っている必要はない。

- タイトルと簡単なメタ情報（生成日時、対象範囲など）
- セクション分け（既存の見出し構造があればそれに従う、無ければ内容から
  自然な区切りを提案する）
- サマリー指標（3〜5個の数値に自然にまとまる場合のみ）
- 重要度・ステータスなど分類できる情報（バッジ化の候補）
- 表・コードブロック・長い補足など、専用コンポーネントが要りそうな箇所

内容が薄くコンポーネントを無理に使う必然性がない場合は、見出しと段落だけの
シンプルなHTMLで十分。

### Step 2: コンポーネントを選んで組み立てる

`references/components.md` を読み、Base（CSS変数・タイポグラフィ）は必ず
使う。それ以外（summary cards / badge / callout / table / code block /
collapsible / toc）は Step 1 で必要と判断したものだけを取り込む。

- 使わないコンポーネントのCSSは含めない（不要なスタイルを毎回貼らない）。
- コンポーネントの見た目・配色はそのまま使ってよいが、内容に応じて
  クラスの組み合わせ・粒度は自由に調整してよい。
- ユーザー入力やログなど信頼できないテキストをそのまま埋め込む場合は
  `<`, `>`, `&` をエスケープする（構造を壊さないため）。

### Step 3: 出力先を決めて保存する

- ユーザーが出力先を指定していればそれに従う。
- 指定がなければカレントディレクトリに `report-<slug>-<YYYYMMDD>.html`
  として保存する（`<slug>` は内容から推測した短い英数字）。
- 一時的な確認用途（「tmpに置いて」等）の場合は、セッション固有の長い
  scratchpadパスではなく `/tmp/report-<slug>-<YYYYMMDD>.html` のような
  短いパスに保存する。パスは後で開いたり共有したりする際に打ちやすいことを優先する。
- 既存ファイルを上書きする可能性がある場合は先に確認する。

### Step 4: 開いて検証する

- 生成したHTMLのタグ構造が閉じているか（未閉タグ・入れ子崩れがないか）目視確認する。
- 保存したら、指示がなくても自動的にブラウザで開く。環境に応じて使い分ける。
  - WSL2: `explorer.exe "$(wslpath -w <path>)"`
  - Linuxデスクトップ: `xdg-open <path>`
  - macOS: `open <path>`
- 応答の最後に、保存先の絶対パスと `file://` URL（ブラウザにそのまま貼れる形式）を必ず提示する。

## デザイン原則

- **自己完結**: 外部CDN・Webフォント・外部画像への参照は使わない。すべて
  インラインCSSで完結させる。
- **テーマ対応**: `prefers-color-scheme` でライト/ダークを自動切り替えする。
  色はCSS変数で管理し、個別要素にベタ書きしない。
- **レスポンシブ**: テーブルやコードブロックなど横に長くなりうる要素だけ
  `overflow-x: auto` で囲む。本文全体を横スクロールさせない。
- **過剰装飾を避ける**: カード・バッジ・ハイライトは情報の意味づけに使う。
  装飾のための装飾、内容量に見合わない演出は避ける。
